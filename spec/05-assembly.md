# 05 — Сборка метаданных с максимальным покрытием

## 5.1 Модель индексации Apple (базис всех правил)

1. Индексируется **объединение слов** из title + subtitle + keyword field (+ бренд, + название категории — их слова дублировать бессмысленно).
2. Поисковый запрос матчится, если все его слова есть в объединении. **Слова из разных полей комбинируются** — фразу `sleep tracker` покрывают `sleep` в subtitle и `tracker` в keyword field.
3. Вес поля в ранжировании: **title > subtitle > keyword field**. Позиция слова внутри поля значения не имеет.
4. Повтор слова в двух полях НЕ усиливает — только сжигает символы. Отсюда золотое правило: **ни одно слово не повторяется между title, subtitle и keyword field** (и не дублирует слова бренда).
5. Для английского Apple матчит единственное/множественное число — хранить только одну форму (правило фолдинга ниже). Для других языков форм не фолдить (консервативно).

## 5.2 Задача оптимизации (формально)

Дано: множество фраз Φ = все кейворды с R ≥ 1 и Score > 0; словарь W = все слова фраз Φ минус стоп-слова минус слова бренда.
Найти: подмножество слов S ⊆ W и его разбиение на (title-слоган, subtitle, keyword field), максимизирующее

```
Σ по фразам p, покрытым S:  Score(p) × PlacementWeight(p)
```

при бюджетах: слоган ≤ `limits.title − len(brand) − 3` (разделитель ` - `), subtitle ≤ 30, keywords ≤ 100.

`PlacementWeight(p)` = вес самого слабого поля, в котором лежат слова p: все слова в title → 1.0; худшее слово в subtitle → 0.85; худшее в keyword field → 0.7. (Константы позиционного веса: title 1.0 / subtitle 0.85 / kwfield 0.7 — захардкожены, в конфиг не выносятся.)

Фраза p **покрыта**, если каждое её слово (после фолдинга) ∈ S ∪ словам бренда. Стоп-слова из фраз при проверке покрытия игнорируются (`habit tracker for adhd` покрывается словами habit, tracker, adhd).

## 5.3 Фолдинг форм (правило дедупликации слов)

Только для `semanticLanguage` = `en*`. Ключ фолдинга вычисляется по правилам СТРОГО в этом порядке (первое сработавшее — финал):

1. Длина слова < 4 ИЛИ слово в `EXCEPTIONS` → ключ = слово как есть.
2. Кончается на `ss` → как есть (`chess`, `class`, `press`, `business`, `access` — не множественное).
3. Кончается на `us` или `is` → как есть (`focus`, `status`, `bonus`, `virus`, `analysis`, `basis`; редкие потери склейки вроде `menus/menu` — допустимая цена).
4. Кончается на `ies` и длина ≥ 5 → `ies`→`y` (`stories→story`, `categories→category`).
5. Кончается на `ches`/`shes`/`xes`/`zes`/`ses`/`oes` → убрать `es` (`boxes→box`, `watches→watch`, `heroes→hero`).
6. Кончается на `s` → убрать `s` (`games→game`, `notes→note`, `planes→plane`, `habits→habit`).

`EXCEPTIONS` — только настоящие ловушки, где правила 2–6 дали бы ложную склейку с другим реальным словом или порчу неизменяемого слова: `news` (иначе склеится с `new`), `lens`, `ios`, `css`, `gps`, `sms`, `canvas`, `atlas`. Список — константа рядом с кодом, с комментарием «сюда добавлять ТОЛЬКО при доказанной ложной склейке, не для красоты».

**Асимметрия ошибок (обоснование дизайна):** пропущенная склейка (`menus` и `menu` остались оба) стоит несколько символов бюджета; ложная склейка (`planes` слился с `plan`) молча выкидывает рабочий кейворд из индекса. Поэтому правила консервативные: сомневаемся — не стрижём. Из слов с одним ключом в S хранится форма с максимальной суммой Score фраз, где она встречается. Для не-en языков фолдинг выключен полностью (каждая форма — отдельное слово).

## 5.4 Алгоритм отбора слов (детерминированный жадный)

```
S = ∅; budgetTotal = слоган + 30 + 100 (символьные бюджеты трёх полей)
повторять:
  для каждого слова w ∈ W \ S:
    gain(w)  = Σ Score(p) по фразам p, которые становятся покрытыми при S ∪ {w}
             + 0.2 × Σ Score(p)/rem(p) по непокрытым p, где w сокращает остаток слов (rem = сколько слов p ещё не в S)
    cost(w)  = len(w) + 1                    // +1 — разделитель (запятая или пробел)
  выбрать w* = argmax gain(w)/cost(w); тай-брейки: больший gain → короче слово → алфавит
  если gain(w*) = 0 → выход из цикла
  если cost(w*) не влезает в суммарный остаток бюджетов → пропустить w (пометить), продолжить
  S += w*
```

Слагаемое 0.2 — «аванс» за частичное продвижение многословных фраз; без него жадный никогда не начнёт собирать трёхсловные запросы. Алгоритм обязан быть чистой функцией `selectWords(phrases, config) → orderedWords` и покрыт юнит-тестами на фикстурах.

## 5.5 Размещение по полям

1. Отсортировать S по вкладу: `contribution(w) = Σ Score(p)` по покрытым фразам, содержащим w.
2. Заполнять по порядку: сначала слоган title (пока влезает), затем subtitle, остаток — keyword field.
3. **Связность:** если топ-слово по вкладу входит в топ-фразу вместе с другим словом, стараться класть их в одно поле (перестановка внутри бюджета допустима, если не меняет множество S) — фраза целиком в title ранжируется сильнее, чем разорванная по полям (PlacementWeight это учитывает: пересчитать варианты и выбрать больший суммарный вес; вариантов мало, полный перебор перестановок топ-6 слов допустим).
4. Результат функции `assemble()` (сохраняется в state, показывается в UI):
```json
{
  "titleWords": ["habit", "tracker"],
  "subtitleWords": ["streak", "routine", "daily"],
  "keywordFieldDraft": "adhd,planner,reminder,focus,goal,builder,motivation,discipline",
  "budgets": { "titleSloganMax": 19, "subtitleMax": 30, "keywordsMax": 100 },
  "coverage": { "phrasesCovered": 61, "scoreCovered": 2140, "scoreTotal": 2610, "coveredShare": 0.82 },
  "topUncovered": [ { "keyword": "...", "score": 38, "missingWords": ["..."] } ]

```

## 5.6 Роль LLM после отбора слов (вызов `phrase`, контракт в 06)

Код выбирает СЛОВА; LLM превращает их в человеческий текст:
- **Title:** `«<Brand> - <слоган>»` — слоган обязан содержать все `titleWords` ровно в выданных формах; порядок и добавление стоп-слов — на усмотрение LLM (стоп-слова не индексируются, но едят бюджет — добавлять только ради читабельности). Пример: words `habit, tracker` → `Habits - Habit Tracker` ЗАПРЕЩЁН (повтор), верно: `Brandname - Habit Tracker`.
- **Subtitle:** фраза из всех `subtitleWords`, читабельная, продающая (subtitle видят люди на странице — это и конверсия, не только индекс).
- **Keyword field:** черновик кода принимается как есть; LLM его не трогает (замены — только через ручное исключение кейвордов пользователем и пересборку).

Ответ `phrase` валидируется кодом (правила 5.7); нарушения возвращаются в промпт повторного вызова (до 3 попыток, см. `04.2`).

## 5.7 Валидация (функция `validate()`) — жёсткие правила, каждое с кодом

| Код | Правило |
|---|---|
| T1 | len(title) ≤ 30; начинается с `brand` + ` - ` |
| T2 | title содержит все titleWords (по фолдинг-ключам) |
| S1 | len(subtitle) ≤ 30; содержит все subtitleWords |
| K1 | len(keywords) ≤ 100; формат: слова через запятую, без пробелов, без пустых элементов |
| X1 | нет повторов фолдинг-ключей между title, subtitle, keywords и словами бренда |
| X2 | нет стоп-слов в keyword field |
| X3 | нет чужих брендовых имён (проверка по списку конкурентов из aso-context.md) |
| W1 | предупреждение: keyword field короче 92 символов — бюджет недобран |

Результат — список нарушений (пустой = зелёный) + coverage-отчёт (какие топ-фразы покрыты каким полем); всё сохраняется в state и отображается на вкладке «Сборка» (`07`).

## 5.8 Спекулятивная добивка keyword field

Если после размещения S бюджет keyword field недобран (W1): добить словами из `unsuggested`-кейвордов с R = 3 (спрос не подтверждён подсказками, но релевантность максимальна — бесплатный лотерейный билет), в порядке убывания R, затем длины по возрастанию. Пометить такие слова в state как `speculative: true` (UI показывает их отдельным цветом).

## 5.9 Кросс-локализация — вторая корзина (входит в v1)

Apple индексирует для storefront'а не одну локализацию: например, для US индексируются en-US **и es-MX**, для RU — ru **и en-GB** и т.д. Это второй полноценный комплект 30+30+100 символов, который почти все ленятся заполнять. Мы заполняем автоматически.

**Алгоритм — второй проход того же жадного отбора (5.4):**
1. Универсум прохода 2: фразы с Score > 0, НЕ покрытые словами прохода 1 ∪ слова бренда.
2. Словарь прохода 2: слова этих фраз минус стоп-слова, минус слова бренда, **минус все слова, выбранные в проходе 1** (слово уже в индексе — дублировать его во второй локализации бессмысленно, это правило 5.1-4, распространённое на обе корзины).
3. Тот же greedy `selectWords`, те же бюджеты (слоган = 30 − len(brand) − 3, subtitle 30, keywords 100), то же размещение (5.5).
4. LLM-вызов `phrase` для второй корзины — отдельный, с пометкой локали (текст должен быть читабельным на языке доп. локализации, где это возможно; при англоязычной семантике в es-MX допустим англоязычный слоган — это штатная практика кросс-локализационного хака).

**Таблица доп. локализаций** хранится рядом со storefront-кодами (`src/apple/storefronts.json`, поле `extraLocale`): us→es-MX, gb→en-AU(→проверить при сборке), ru→en-GB, de→en-GB, fr→en-GB и т.д.; если для страны доп. локаль неизвестна — проход 2 пропускается с событием в журнал. Конфиг прогона: `"extraLocale": true` (дефолт; выключаемо в расширенных настройках).

**Валидация:** правила T/S/K/X/W применяются к каждой корзине отдельно, плюс новое правило **X4: ни один фолдинг-ключ не повторяется между корзинами** (включая бренд). Coverage-отчёт считается по объединению обеих корзин; `topUncovered` — то, что не влезло даже во вторую (остаётся подсказкой в UI).
