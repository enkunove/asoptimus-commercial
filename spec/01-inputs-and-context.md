# 01 — Вход, бизнес-контекст, конфигурация, хранилище

## 1.1 Единственный вход: бриф

Пользователь загружает **один файл `.md` или `.txt`** (или вставляет текст в textarea — сохраняется как файл) через дашборд при создании прогона. Требования:

- кодировка UTF-8, размер 200 символов … 200 КБ;
- меньше 200 осмысленных символов → форма не даёт создать прогон и объясняет, что дописать (см. подсказку под полем: «что делает апка, для кого, конкуренты, рынок»);
- язык брифа любой — оркестратор работает с любым, но семантика генерируется на языке целевого рынка из настроек прогона.

Файл копируется в директорию прогона как `brief.md`. Никаких других способов входа (репозиторий, URL, App Store ID) в v1 нет.

## 1.2 Извлечение бизнес-контекста

Первый шаг прогона — LLM-вызов `context` (контракт в `06-llm-adapters.md`): бриф → структурированный контекст:

```json
{
  "productSummary": "одним абзацем",
  "category": "Health & Fitness",
  "jobsToBeDone": ["..."],
  "audience": "кто ищет и какими словами думает",
  "featureVocabulary": ["habit tracker", "streak", "..."],
  "competitors": ["Sleep Cycle", "..."],
  "antiSemantics": "чем апка НЕ является; по каким словам НЕ индексироваться",
  "targetLanguage": "en"
}
```

Контекст показывается пользователю на экране прогона **до старта цикла** (шаг «Контекст» с кнопками «Выглядит верно, поехали» / «Отредактировать»). Редактирование — прямо в UI по полям; отредактированная версия сохраняется и используется во всех последующих промптах. Это единственная обязательная точка подтверждения человеком за весь прогон.

## 1.3 Конфигурация прогона

Задаётся в форме «Новый прогон» (разумные дефолты, всё редактируемо; после старта прогона — read-only). Хранится как `config.json` в директории прогона:

```json
{
  "brand": "Somna",               // обязателен; идёт в title до " - "
  "country": "us",                // storefront из таблицы 02
  "semanticLanguage": "en",       // язык генерации гипотез; дефолт = основной язык страны, выбирается в форме
  "language": "en_us",            // lang для search API; выводится из country+semanticLanguage
  "sampleSize": 150,              // точка остановки (форма: слайдер 50–500)
  "batchSize": 20,
  "exploreRatio": 0.3,
  "improvementRounds": 2,
  "serpTop": 10,
  "model": "claude-opus-4-8",     // модель адаптера для этого прогона (список в 06)
  "extraLocale": true,            // вторая корзина кросс-локализации (05.9)
  "weights": {
    "popularity":  { "depth": 0.7, "rank": 0.3 },
    "difficulty":  { "volume": 0.45, "quality": 0.15, "freshness": 0.15, "match": 0.25 },
    "opportunity": { "popularityExp": 0.6, "easeExp": 0.4 }
  },
  "limits": { "title": 30, "subtitle": 30, "keywords": 100 },
  "http": { "requestsPerMinute": 18, "cacheTtlDays": 7, "timeoutMs": 10000, "retries": 3 },
  "stopwords": ["app", "apps", "free", "best", "top", "new", "a", "an", "the", "and", "or", "for", "of", "with", "your", "my", "&"]
}
```

Валидация (при создании прогона, ошибки показываются у полей): `brand` непустой и `len(brand)+3 <= limits.title`; `sampleSize` в [30, 500]; `batchSize` в [5, 50]; `exploreRatio` в [0,1]; суммы весов popularity и difficulty = 1.0 (±0.001); `country` — из таблицы storefront-кодов.

В форме продвинутые поля (weights, http, stopwords) спрятаны под «Расширенные настройки» — обычному пользователю достаточно brand / country / **язык семантики** / sampleSize / model.

**Язык семантики** — селект в основной части формы; при выборе страны автоматически подставляется её основной язык (us→en, ru→ru, de→de, mx→es…; таблица — рядом со storefront-кодами), но пользователь может выбрать любой другой: например, испанскую семантику для US-стора (испаноговорящая аудитория ищет по-испански прямо в американском сторе). Валидация всё равно идёт по подсказкам выбранного storefront — фразы на языке, которым в этом сторе не ищут, честно получат P=0 и отсеются. Все LLM-задачи (`06.3`) получают `semanticLanguage` и генерируют/оценивают на нём.

## 1.4 Хранилище данных приложения

Всё лежит в **директории данных** (переопределяется флагом `--data-dir`):

- macOS/Linux: `~/.aso-util/`
- Windows: `%APPDATA%\aso-util\`

```
~/.aso-util/
├── auth.json                 # креды провайдеров (chmod 600; формат в 06)
├── settings.json             # глобальные настройки: активный провайдер, порт
├── cache/                    # HTTP-кэш Apple — ОБЩИЙ для всех прогонов
│   └── <sha1>.json
└── runs/
    └── somna-us-2026-07-15/  # slug: brand-country-date(-N при коллизии)
        ├── brief.md
        ├── config.json
        ├── context.json      # результат вызова context (+ правки пользователя)
        ├── state.json        # все кейворды, метрики, фаза, финальные поля
        ├── llm-log.jsonl     # журнал LLM-вызовов: промпт, ответ, usage (07/06)
        └── events.jsonl      # журнал событий пайплайна для ленты UI
```

Кэш Apple общий: две апки в одной нише переиспользуют подсказки и выдачу. `state.json` пишется атомарно (temp + rename); убийство процесса не теряет данные, прогон возобновляем с места остановки.

## 1.5 Мультистрановость

Один прогон = один storefront. Для второй страны пользователь создаёт новый прогон с тем же брифом (форма «Новый прогон» предлагает «клонировать из существующего» — копирует бриф, контекст и конфиг с другой страной).
